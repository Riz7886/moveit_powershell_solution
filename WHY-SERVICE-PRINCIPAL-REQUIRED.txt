WHY SERVICE PRINCIPAL IS REQUIRED


FOR YOUR CLIENT EXPLANATION


THE QUESTION

Why do we need to create a Service Principal in Azure for Datadog monitoring


THE SIMPLE ANSWER

Datadog cannot read Azure metrics without Azure permissions

Service Principal gives Datadog permission to read Azure metrics

Without Service Principal, Datadog cannot see anything in Azure


THE TECHNICAL EXPLANATION

Datadog is a third-party monitoring tool outside of Azure

Azure does not allow external systems to access data by default

Service Principal is an identity that Azure recognizes

It acts like a user account but for applications

We give this identity Monitoring Reader permission

This allows Datadog to read metrics from Azure

Without this, Datadog has no way to access Azure data


HOW IT WORKS

Step 1: Create Service Principal in Azure
This creates an identity that Datadog can use

Step 2: Assign Monitoring Reader role
This gives the identity permission to read metrics

Step 3: Give credentials to Datadog
Datadog uses these credentials to authenticate with Azure

Step 4: Datadog reads metrics
Now Datadog can pull CPU, memory, disk, network data from Azure

Step 5: Monitors evaluate data
Monitors check if metrics exceed thresholds and alert


WHY WE CANNOT SKIP THIS

Option 1: No Service Principal
Result: Datadog cannot access Azure
Result: All monitors show NO DATA forever
Result: No monitoring, no alerts, no visibility

Option 2: With Service Principal
Result: Datadog can access Azure
Result: Monitors receive data
Result: Full monitoring and alerting works


COMPARISON TO OTHER INTEGRATIONS

AWS Integration: Requires IAM Role (same concept as Service Principal)
Google Cloud: Requires Service Account (same concept)
Azure: Requires Service Principal

All cloud providers require this security mechanism

This is industry standard for third-party integrations


SECURITY BENEFITS

Service Principal has limited permissions
It can only READ metrics, cannot modify anything
It cannot create, delete, or change Azure resources
It cannot access sensitive data
It only sees performance metrics

This is more secure than giving full access


ALTERNATIVES THAT DONT WORK

Option 1: Use your personal account
Problem: Security risk, account could be disabled
Problem: Violates least privilege principle
Problem: Not suitable for production

Option 2: Use API keys only
Problem: Azure does not allow this
Problem: Datadog needs Azure credentials to authenticate
Problem: No way around Service Principal requirement

Option 3: Install agents on all VMs
Problem: Does not provide Azure-level metrics
Problem: Cannot monitor SQL, Storage, App Services
Problem: More complex, higher maintenance
Problem: Still need Service Principal for full visibility
